# CLAUDE.md

This file provides guidance to Claude Code when working with code in this repository (or similar projects based on observed patterns).

## Observed Patterns from Analyzed Projects

### Common Tech Stack
- **Framework**: Next.js (with React 18) or Create React App (CRA)
- **Language**: TypeScript (strict mode)
- **Styling**: Tailwind CSS, Chakra UI, Material-UI (MUI), SCSS/SASS
- **State Management**: 
  - React Query (for server state)
  - Zustand (lightweight client state)
  - Redux Toolkit (for complex global state)
- **Data Fetching**: Axios (REST), React Query (for caching & background updates)
- **Forms**: React Hook Form (with Zod/Yup validation) or Formik
- **UI Components**: 
  - Headless/UI libraries: Chakra UI, MUI
  - Animations: Framer Motion
  - Toast: React Toastify
- **Development Tools**:
  - Linting: ESLint (with `eslint-config-next` or custom configs)
  - Formatting: Prettier (with plugins like `prettier-plugin-tailwindcss`)
  - Testing: Jest, React Testing Library, Cypress (in some), Storybook for UI components
  - CI/CD: GitHub Actions (via `gh-pages` for deployment to GitHub Pages)
- **Build Tools**: Vite (for CRA-like setups), Next.js built-in, Webpack (via CRA/Storybook)

### Common Project Structure
```
src/
├── components/           # Reusable UI components
├── pages/                # Next.js routes (or route equivalents)
├── src/                  # (in CRA) source code
│   ├── modules/          # Feature-based modules (e.g., create-post-module)
│   ├── services/         # API instances, hooks, utils
│   ├── store/            # State management slices (Zustand, Redux)
│   ├── styles/           # Global CSS, SCSS, Tailwind base
│   └── utils/            # Helper functions, constants
├── public/               # Static assets
├── tests/                # Test files (if separated)
├── .storybook/           # Storybook configuration
└── integration/          # Integration tests (e.g., Cypress + Storybook)
```

### Typical npm/yarn/pnpm Scripts
- `dev` – start dev server (Next.js dev, CRA start, or Vite)
- `build` – production build
- `start` – start production server
- `lint` – run ESLint
- `format` – run Prettier (if configured)
- `test` – run Jest
- `test:ui` – launch Storybook
- `build-storybook` – build Storybook static site
- `predeploy` / `deploy` – build and push to GitHub Pages (via `gh-pages`)

### Key Configuration Files
- `next.config.js` / `next.config.mjs` – Next.js customization
- `tailwind.config.cjs` – Tailwind setup (with plugin for prettier)
- `tsconfig.json` – TypeScript strictness, paths
- `.eslintrc.json` – ESLint rules (often extends `eslint-config-next` or custom)
- `.prettierrc` / `prettier.config.cjs` – Prettier options
- `yarn.lock` / `package-lock.json` / `pnpm-lock.yaml` – lockfiles
- `.gitignore` – standard Node.js ignores + `.env`, `build/`, `.next/`, `coverage/`, `storybook-static/`

### Development Workflow Notes
1. **Environment Variables**: Use `.env.local` for secrets (never commit). Loaded automatically in Next.js/CRA.
2. **Code Splitting**: Next.js does this automatically; for CRA, use `React.lazy` with Suspense.
3. **Asset Optimization**: Next.js optimizes images via `<Image>`; ensure `next.config.js` includes domains if needed.
4. **Storybook**: Run `npm run storybook` to develop UI components in isolation; stories live alongside components (`[ComponentName].stories.tsx`).
5. **Testing**:
   - Unit tests: Jest + React Testing Library (`*.test.tsx`)
   - Integration tests: Cypress or Playwright (see `integration/` folder)
   - Visual regression: `jest-image-snapshot` (used in Todolist2)
6. **Deployment**: 
   - GitHub Pages: `npm run predeploy` builds, `npm run deploy` pushes `build/` to `gh-pages` branch.
   - For Next.js: can also deploy to Vercel (default) or Docker.

### Tips for Working with Claude Code in These Projects
- When adding a new feature, consider placing it in a feature folder under `modules/` or `src/features/`.
- For API calls, create a reusable `axiosInstance` in `services/` and extend with interceptors if needed.
- Prefer React Query for server state; use Zustand or Redux Toolkit for client-only state.
- Keep components small and composable; use UI library primitives (Chakra, MUI) rather than rebuilding from scratch.
- When styling, adopt utility-first (Tailwind) or theme-based (Chakra/MUI) approach; avoid inline styles unless dynamic.
- Always run `lint` and `test` before committing; CI will catch issues, but local feedback is faster.
- If you see `.storybox`, run `storybook` to view/edit components in isolation.
- For debugging state, consider adding `zustand/middleware/logger` or Redux DevTools (already configured in Redux Toolkit).

## При клонировании нового репозитория:

  # Вместо HTTPS:
  # git clone https://github.com/lunia56/имя-репо.git

  # Используйте SSH:
  git clone git@github.com:lunia56/имя-репо.git

  Для уже существующего локального репозитория:

  # Проверьте текущий remote
  git remote -v

  # Если он использует HTTPS, измените на SSH:
  git remote set-url origin git@github.com:lunia56/имя-репо.git

  Затем просто пушьте:

  git push origin main  # или вашу ветку
